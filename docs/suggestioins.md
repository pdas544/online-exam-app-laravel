# Project review: inconsistencies and optimization ideas

## Inconsistencies / risk areas
- Public access exposure: resource routes for `users`, `subjects`, `exams`, and `questions` are declared outside the `auth` group, and `exams` is also registered twice. Consider moving all resource routes inside the authenticated block and removing duplicates to avoid unauthenticated access and route conflicts. See [routes/web.php](routes/web.php).
- Auth checks in constructors: `ExamController` and `StudentDashboardController` call `Auth::user()` in the constructor without guaranteed `auth` middleware, which can be `null` and raise errors. Prefer middleware-based authorization or guard checks inside methods. See [app/Http/Controllers/ExamController.php](app/Http/Controllers/ExamController.php) and [app/Http/Controllers/Dashboard/StudentDashboardController.php](app/Http/Controllers/Dashboard/StudentDashboardController.php).
- Commented middleware: `ExamSessionController` and `StudentDashboardController` have `auth` middleware commented out, which weakens access control compared to the route-level intent. See [app/Http/Controllers/ExamSessionController.php](app/Http/Controllers/ExamSessionController.php) and [app/Http/Controllers/Dashboard/StudentDashboardController.php](app/Http/Controllers/Dashboard/StudentDashboardController.php).
- Missing model reference: `ExamSession::violations()` references `ViolationLog`, but no model exists in [app/Models](app/Models). Add the model (and relations to `User`/`Exam`) or update the relation to the correct class. See [app/Models/ExamSession.php](app/Models/ExamSession.php) and [database/migrations/2026_02_23_144737_create_violation_logs_table.php](database/migrations/2026_02_23_144737_create_violation_logs_table.php).
- Broadcast event import: `AnswerSaved` uses `Channel` but does not import it, which will trigger a class not found error at runtime. See [app/Events/AnswerSaved.php](app/Events/AnswerSaved.php).
- Inconsistent filtering logic: `ExamController::index()` checks `academic_year` but reads `$request->search` for year/semester filtering, which looks like a bug and can produce unexpected results. See [app/Http/Controllers/ExamController.php](app/Http/Controllers/ExamController.php).
- Realtime config usage: `TeacherMonitor` uses `this.config.teacherId` but never initializes `this.config`, which can break websocket subscriptions. See [resources/js/teacher-monitor.js](resources/js/teacher-monitor.js).

## Optimization opportunities
- Reuse `StudentAnswer::autoGrade()`: `submit()` currently handles only unanswered items and manual score aggregation; consider calling `autoGrade()` for each answer to centralize grading rules and reduce duplicated logic. See [app/Models/StudentAnswer.php](app/Models/StudentAnswer.php) and [app/Http/Controllers/ExamSessionController.php](app/Http/Controllers/ExamSessionController.php).
- Reduce query count on submission: `submit()` iterates `answers` without eager-loading; loading `answers` once and using in-memory operations can reduce repeated DB calls. See [app/Http/Controllers/ExamSessionController.php](app/Http/Controllers/ExamSessionController.php).
- Centralize authorization: controllers manually call `abort(403)` in several places. Consider using policies or gate middleware to keep authorization consistent and testable. Examples in [app/Http/Controllers/ExamController.php](app/Http/Controllers/ExamController.php) and [app/Http/Controllers/QuestionController.php](app/Http/Controllers/QuestionController.php).
- Consider using `broadcastOn()` defaults consistently: events extend `BaseExamEvent` but `AnswerSaved` overrides channels to `student.*` only. If this is intentional, document it; otherwise reuse the base channels for uniform monitoring. See [app/Events/BaseExamEvent.php](app/Events/BaseExamEvent.php) and [app/Events/AnswerSaved.php](app/Events/AnswerSaved.php).
